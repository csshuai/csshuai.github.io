分享会-2016-03-21

为什么要推倒重来

以前的框架有些坑

  1. Eventbus 坑
  2. Library 框架的坑: 很容易就会发生空指针
  3. 大家写代码中的坑: 有些该回收的东西没有回收, 该关闭的没关闭, 在主线程执行阻塞方法
  4. 代码混乱, 没有统一的代码规范
  5. 弃用的代码没有删除, 弃用的资源没有删除

Fragment

  • 获取数据的请求放入 onCreate(…) 方法中
  • 给 Fragment 发送数据使用 Fragment.setArguments(Bundle) 方法
  • FragmentManager 如何管理 Fragment
  • Fragment.getFragmentManager() 和 Fragment.getChildFragmentManager()

Eventbus

Eventbus 缺陷

  1. 代码不好跟踪
  2. 滥用导致数据错误(目前代码中较多)
    例如: FragmentA 注册了 Eventbus, 如果FragmentA同时启动了2个实例 C 和 D, 在 C 的 mAdapter 中发送了一个消息给 FragmentA, C 和 D 都会接收到消息
  3. 没有注销导致内存溢出

Eventbus 能使用的地方

  • 只用来发送服务的消息, 一定程度上代替广播的作用
  • 其他情况用 Eventbus 需要讨论
  • Eventbus 接收到的消息用 getHandler() 发送出去再次处理, 防止空指针

StrictMode

PausedHandler

RxJava

Retrofit2

proguard-annotations

代码规范(提交到 git 的代码必须符合以下要求)

  • 注释的代码块必须删除, 并在 Commit Message 中做必要的说明
  • Google Java编程风格指南
  • 废弃的功能必须删除
  • 废弃的资源必须删除
  • 重载的方法放在一起, 且按照生命周期顺序
  • 私有方法放在最后
  • 写 api 时将文档连接放到注释中
  • Adapter 多个 Onclick 事件只需要一个 OnClickLi\stener
  • 代码不要有太多空格(方法之间只要一个空格)
  • 共用 value 才放入 values 文件夹, 否则直接在代码里写

工作规范

  • 下班前提交代码
  • 提交的代码必须确认 app 能跑起来
  • 报告量化工作
    量化的目的: 找到短板, 寻找解决方案, 提升自身价值.
    1. 总共有 xx 个 bug, 其中 xx 个是需求不清晰, xx 个是没理解需求, xx 个编码问题. 修复xx个 bug. 怎么样减少 bug?
    2. 开发了2个功能, 功能1开发进度 xx%, 花费时间多少. 功能2开发进度 xx%, 花费时间多少. 有什么心得体会?
    3. 代码检查: xx 个功能 bug, xx 个风格不符, xx 性能问题. 怎么样减少问题?
  • 上下班前确认需求设计的变更
  • 2-3人交叉工作
  • 下班前半小时: 提交代码, 总结今天的得失, 思考怎么样进行改进, 有收获向大家分享

其他问题

  • 网络请求提交数据时不要启用重试机制??
  • mipmap 只是用存放 ic_launcher
  • xml 图形资源放入 drawable 文件夹
  • 删除未使用的资源: 右键 Refactor -> Remove Unused Resources…

新版本注意事项

  • 色值: 共用色值写入 colors.xml, 不共用色值直接写在代码中
  • 尺寸值: 共用尺寸值写入 dimens.xml, 不共用尺寸值接写在代码中
  • 以上如果需要做适配, 可以写在 xml 中
  • TextView 和 EditView 不要设置 cursorDrawable. 如果是默认字体色(#333)和 Hits 色(#999), 也不要设置色值
  • 图片资源的问题: 优先从 svn 中复制图片, 并按#资源命名规则 重新命名 (不要从原项目中复制)
  • 自定义的一些辅助类 bean 不要放入 bean 包中, bean 包中全部都是服务端业务需要的类, bean 中不要加入非必要(客户端不用)的字段
  • 使用图片和颜色状态来代替在代码中动态设置
  • glide.with(xxx), 使用范围最小的参数
  • 输入法统一使用 resize 的方式
  • 分模块. 例如充电模块, 有 N 多的功能就写在一个 Fragment 中, 要分成多个 Fragment
  • emojicon 使用 EmojiconHandler 去处理 Emoji 表情, 不要直接使用 EmojiconEditText 和 EmojiconTextView
  • 使用 TabLayout, 不要使用 UnderlineTabPageIndicator
  • Rxjava 使用场景

  • recyclerview 不要用隐藏的方式, 要重写 getItemCount 方法

  • 列表播放视频使用 VideoPlayerManager

框架讲解

资源命名规则

  • <类型…>_<限定词…>, 在能表达意思的基础上, 越短越好

  • 资源文件名只能使用小写字母数字下划线(a-z0-9_), 且只能以小写字母开头

  • layout|menu 命名

    1. Fragment 的 onCreateContentView 中的 layout 命名: content<类型…><限定词…>
    2. Fragment 的 onViewCreating 中的 layout 命名: fragment<类型…><限定词…>
    3. Activity 的 layout 命名: activity<类型…><限定词…>
    4. Adapter 的 layout 命名: item<类型…><限定词…>
    5. Dialog 的 layout 命名: dialog<类型…><限定词…>
  • drawable 命名
    1. 图标: ic<类型…><限定词…> 例如: ic_toolbar_back_selector
    2. 背景: bg<类型…><限定词…>
  • color 命名
  • value 命名
    1. 某个库用的 Value 单独给放入一个文件中 例如: values_tab_page_indicator

Android 技术分享(一)

Activity

Activity 是一个应用组件，用户可与其提供的屏幕进行交互，以执行拨打电话、拍摄照片、发送电子邮件或查看地图等操作。 每个 Activity 都会获得一个用于绘制其用户界面的窗口。窗口通常会充满屏幕，但也可小于屏幕并浮动在其他窗口之上。

一个应用通常由多个彼此松散联系的 Activity 组成。 一般会指定应用中的某个 Activity 为“主” Activity，即首次启动应用时呈现给用户的那个 Activity。 而且每个 Activity 均可启动另一个 Activity，以便执行不同的操作。 每次新 Activity 启动时，前一 Activity 便会停止，但系统会在堆栈（“返回栈”）中保留该 Activity。 当新 Activity 启动时，系统会将其推送到返回栈上，并取得用户焦点。 返回栈遵循“后进先出”堆栈机制，因此，当用户完成当前 Activity 并按“返回” 按钮时，系统会从堆栈中将其弹出（并销毁），然后恢复前一 Activity。（任务和返回栈文档中对返回栈有更详细的阐述。）
